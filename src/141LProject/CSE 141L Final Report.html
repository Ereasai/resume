  1. What is special about our machine?
    1. Fixed-length: The ISA has a fixed length of 9 bits for each instruction.
    1. RISC: The ISA has a relatively small set of simple, general-purpose instructions, with few specialized ALU operations for 3 required programs.
    1. Load-Store Architecture: The machine supports separate load and store instructions to move data between memory and registers, and the arithmetic and logic operations are performed only on registers.
    1. Register-based: Most operations use registers for their operands. This means that the operands are stored in registers rather than embedded in the instruction.
    1. Operand Registers: To expand our instruction set, we broke up some instructions into multiple steps. Instead of embedding register operands inside the machine code, we load the operands into an Operand Register and perform arithmetic.
    1. Variable Opcode Length: Some instructions require us to embed immediate values in the machine code, which means having more bits will be helpful. Some instructions only require 3 bits of the operand, which means that we have a lot of waste bits for the machine code. We optimized the machine code in a way that instructions with immediate values benefit a lot from short opcodes (2-3 bits), while those that only require little bits are encoded with long opcodes (>3).
    1. Overall, the machine is a RISC, load-store, register-based architecture.

3. Machine Specification

  1. Instruction Format

    Our instructions do not fall into a certain category—or it should not be thought of that way since it wasn’t designed that way. To deal with the lack of bits in each instruction, we tried to optimize our machine code so that:

    • the length of the opcode is short for those that require more bits in the operand (for instance, instructions with immediate values benefit from having more bits for their operand).
    • the length of the opcode is longer for those that do not require that many bits (see, load, and store).

    Below is a binary tree that shows how each instruction is formatted. Notice that as we go deeper down the tree, fewer operand bits are available. Hence, instructions that benefit from more operand bits are placed closer to the root node b0, while ones that have either fixed length or fewer operand bits are less prioritized.

    • If a cell is marked as either 0 or 1, then that index of III﻿ must be that bit.
    • If a cell is marked as X, it means that those bits are ignored by the control unit.
    • If a cell is marked as something else, like RdR_dRd​﻿, it means that those bits are allocated to represent that argument.
      • For instance, RdR_dRd​﻿ is given 3 bits to represent the source/destination general purpose register.

    1. What type of branches are supported? Absolute branching is supported.
    1. how are the target addresses calculated? The target address is [A5...0∣∣B5...0][A_{5...0} || B_{5...0}][A5...0​∣∣B5...0​]﻿.
    1. what is the maximum branch distance supported? The total length of the target address is 12 bits, which is enough to cover all instruction memory. Hence the maximum distance will be jumping between 0 to 212−1=40952^{12}-1=4095212−1=4095﻿.
  1. Address models.
    1. What memory addressing modes are supported, e.g. direct, indirect? Indirect addressing only. Technically, you are able to load an immediate value into A/BA/BA/B﻿ and load. However, we are not sure if this still counts as a direct addressing as we are accessing the data at the contents of some register.
    1. How are addresses calculated? Addresses are determined by adding values of AAA﻿ and BBB﻿.

4. Programmer’s Model [Lite]

  1. How should a programmer think about how your machine operates?

    A programmer should think of ALU operations as a 3-step process of loading operands, doing some operation on the operands, then storing it back into a general-purpose register. It’s very similar to the traditional reg-reg ISA, but just extra steps for loading the operands. Other operations are more traditional: it involves moving between A/BA/BA/B﻿ registers, general registers, and data memory; and jumps and conditional jumps. 

    // example of doing an ALU operation.
loadOPI A, #2 // load 2 into A.
loadOPI B, #3 // load 3 into B.
add r0        // store result of adding A and B to r0.

  1. Will your ALU be used for non-arithmetic instructions (e.g., MIPS or ARM-like memory address pointer calculations, PC relative branch computations, etc.)? If so, how does that complicate your design?

    Yes. Address pointers can be thought of as offset + base as we are getting the memory contents at the address [A] + [B]. The ALU also requires a NOOP to pass through some values through ALU (used as a data path). Finally, it is used to execute compare operations, which sets the flag register.

    As a result, it requires us to implement more complicated data paths (more MUXs) and extra control logic for the MUXs, ALU, and other components.

Program Counter

Module file name: PC.sv

Module testbench file name: tb_PC.sv

Functionality Description

This module implements a program counter that tracks the program execution flow. It supports absolute jumps and increments the program counter by 1 for each clock cycle, unless a jump operation is enabled or a reset signal is received.

The module has the following inputs:

  • clk: Clock signal used for synchronous operations.
  • reset: A signal that resets the program counter to zero.
  • jmp_en: A signal indicating whether a jump operation is enabled.
  • jmp_dest: A length PC_W input representing the destination address for a jump operation.

The module has one output:

  • pc_out: Output of length PC_W that represents the current value of the program counter.

Combinational: output the current value of PC to pc_out.

Sequential: if no control signals are enabled, increment PC. If jmp_en is enabled, set the current PC to jmp_dest.

Details

The raw text is first tokenized and converted into instruction objects. The parser will also determine the PC location of each labels. This part is tricky because we have some macros—which means that some assembly code will be converted into multiple lines of machine code. jmp and jif instructions require us to replace its label arguments with a concrete address to jump to. Hence, after the label mapping is ready, modify jmp and jif. Finally, we have a complete list of instructions that can be used to simulate the program. Furthermore, each instruction object can produce machine code based on its arguments.

Most of the parser’s job is just converting each line of assembly instruction into machine code instruction. However, the jump instructions get more complex. Since our ISA requires us to set the operand registers to the jump destination before executing jump, the parser needs to map assembly instructions that have the form jmp <label> into three machine code instructions—two for setting the operand registers, and one for jump.

We might add more macro-like instructions so we (as programmers) can avoid using operand registers while programming.
